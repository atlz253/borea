# Borea Aesthetics

## Core Idea

Borea is a research station for software engineering.

Outside — chaos:

- constantly changing technologies;
- deadlines;
- complex projects;
- massive codebases;
- information noise.

Inside Borea:

- order;
- knowledge;
- automation;
- calm;
- engineering discipline.

The developer does not fight tools — they research and build the product.

## Central Image

Not a futuristic metropolis.

Not cyberpunk.

Not a spaceship.

But an autonomous polar research station.

Imagine:

- night;
- snow;
- strong wind;
- aluminium laboratory buildings;
- warm light from windows;
- quiet hum of equipment;
- computers;
- maps;
- instruments;
- a feeling of total focus.

This is the feeling the interface should evoke.

## Emotions

Borea should be associated not with energy, but with:

- confidence;
- calm;
- reliability;
- focus;
- professionalism.

If most IDEs could be described as:

"Faster!"

Then Borea should say:

"Everything is under control."

## Philosophy

Not "AI will replace the developer."

But:

AI is another researcher at the station.

It helps.

Searches.

Documents.

Checks.

Suggests.

But does not take control.

## Visual Style

Minimalism.

Generous whitespace.

No clutter.

Dark surfaces.

Matte materials.

Cool tones.

Rare warm accents.

## Colors

### Primary

- graphite;
- deep blue;
- ice grey.

### Accents

- polar teal;
- cold blue;
- white.

### Rare

- warm amber light (e.g., successful completion indicators).

### Do not use

- acid colors;
- bright red without necessity;
- neon cyberpunk.

## Interface

Everything should feel like a scientific station control panel.

Cards resemble equipment panels.

Tables look like observation logs.

Change history reads as an expedition journal.

Documentation feels like a research archive.

## Sounds

If they ever appear:

not "beeps".

But:

- quiet mechanical clicks;
- muffled signals;
- sounds of laboratory equipment.

## Illustrations

Not cartoon characters.

Instead:

- maps;
- diagrams;
- topographic lines;
- coordinate grids;
- contour mountain drawings;
- polar stations;
- antennas;
- weather masts;
- compasses.

## Component Naming

Instead of abstract "Module", use expedition-themed names.

For example:

| Component | Name |
|-----------|------|
| Workspace | Station |
| AI | Researcher |
| Knowledge Base | Archive |
| Search | Compass |
| Sync | Fabric |
| Terminal | Console |
| Logs | Journal |
| Documentation | Almanac |
| Monitoring | Beacon |

But do not overplay. The user should not search for "Researcher" when they need an AI agent. Thematic names work best for internal modules, documentation, or marketing materials. In the interface, stick to familiar terminology.

## Logo

Minimal.

Do not draw the Northern Lights.

Instead, use a symbol that simultaneously suggests:

- a snowflake;
- a compass;
- the North Star;
- a stylised letter B.

## Documentation

Documentation should feel like research station records.

Not "Quick Start" — but "First Expedition".

Not "Examples" — but "Field Notes".

Though here, caution is needed. For public documentation, keep familiar section names (Getting Started, Guides, API Reference). Create atmosphere through styling, illustrations, and the language of introductory texts.

## Motto

Several ideas that pair well with the project philosophy:

*Calm software engineering.*

or

*Where software finds clarity.*

or

*Built for focus.*

or

*A calm place to build software.*

or

**Engineering without chaos.**

The last one is especially good. It perfectly reflects the project's core idea.

## Design Principles

These should be part of the project itself:

- **Clarity over cleverness** — clarity matters more than being impressive.
- **Silence over noise** — minimal visual and information noise.
- **Knowledge over repetition** — knowledge should be preserved and reused.
- **Automation over routine** — automate the routine, not the thinking.
- **Systems over tools** — Borea is a coherent system, not a collection of disparate functions.
- **Human remains in command** — AI helps the engineer, but the engineer makes decisions.
